U-OAMM

Utility OT Assurance Maturity Model — derive assumptions → behavior → risk → zones → conduits → controls → assurance questions.

Utilities SAS / Protection / SCADA PPC / Renewables Evidence-first assurance
Core position
Zoning, conduits, and controls are outcomes. If you have them without explicit assumptions, behaviors, and risks, you do not have assurance — you have decoration.
How to use this
  1. Assess maturity level (highest stage you can answer without guessing).
  2. Work sections in order. Skipping a section invalidates downstream work.
  3. Generate assurance questions last. They are the output.
Derivation pipeline (strict order)
Reality → Assumptions → Behavior → Risk → Grouping → Zones → Conduits → Controls → Assurance Questions
Rule Reordering steps produces answers that feel confident and are wrong.
Maturity model

A utility’s maturity is the highest question it can answer without guessing. Controls don’t raise maturity by themselves; they only count if they are derived from explicit risk and tied to evidence.

Level Name What exists What is missing / tell-tales
0 Artifact-driven Tools, policies, diagrams, audits No explicit assumptions/behavior/risk; zoning precedes understanding
1 Assumption awareness Assumptions exist in people Not written; not linked to behaviors; knowledge exits with staff
2 Behavior characterized Required behaviors, states, timing recognized Behavior not yet turned into risk; zones still topology-first
3 Risk derived Risk from behavior deviation, impact described Risk not grouped; zoning not strictly risk-coherent
4 Risk-coherent zoning Zones justified by shared risk and behavior Conduits still implicit or overly broad
5 Conduit-bounded Cross-zone behavior contracts, state/time constraints Controls partially inherited, not fully derived
6 Control-derived implementation Controls mapped to risks; testable and removable Evidence exists but may lack semantics/context
7 Assurance-capable Assurance questions defined; evidence answers them Assumptions not continuously validated
8 Assurance-sustained Assumptions continuously validated; maturity doesn’t regress Rare; requires institutionalization and change management
Classification rule
You are at level N if you can perform all steps up to N using explicit statements and repeatable evidence, not memory and heroics.

Assumptions are conditions that must hold for the system to operate safely as designed. If assumptions are implicit, assurance is impossible.

Assumption classes (utility-specific)
  • Process: physics bounded, deterministic states
  • Functional: protection/control logic correct as engineered
  • Temporal: latency/jitter bounds, sequence expectations
  • Topology: stable wiring, known data paths, controlled changes
  • Human: SOP adherence, role boundaries, maintenance windows
  • Access: physical access controls, trusted consoles
Failure modes
  • Assumption exists only in people (not documented)
  • Assumption is true in normal state but false in maintenance
  • Assumption depends on a vendor feature nobody tested
  • Assumption breaks silently (no detection)
  • Assumption drift over time (config creep)
Assumption extraction method (repeatable)
  1. Start with what exists: assets, links, functions, states
  2. Ask: what must be true for this to be safe?
  3. Write assumptions as testable statements (no vibes)
  4. Tag each assumption with: state, owner, evidence, expiry
  5. Mark assumptions that cannot be validated as high-risk by default
Outputs
  • Assumption register
  • State map (normal/fault/maintenance/commissioning)
  • Assumption-to-function mapping
  • Assumption validation plan (later becomes assurance)

Behavior is an observable interaction or effect needed for a function under a state. In utilities, behavior is state-dependent and time-bounded.

Behavior fields (minimum)
  • Actor → Target
  • Function (what it enables)
  • State (normal/fault/maintenance/commissioning)
  • Timing bounds (latency/jitter/sequence)
  • Frequency bounds
  • Effect on process (visibility/control/protection)
Classification
  • Required: must exist for engineered function
  • Acceptable: optional but controlled
  • Unnecessary: provides no function value
  • Forbidden: creates process risk by default
Utility examples
  • Protection GOOSE inside bay/station: required in fault windows; unnecessary outside unless justified.
  • MMS control: required only when a validated operator workflow exists; otherwise forbidden.
  • Engineering protocol access: required only in maintenance/commissioning with explicit bounds.
Output
Behavior inventory, state-tagged, including what is present today and what should not exist.
Failure modes
  • Behavior described in packet terms only
  • No state distinction (normal vs maintenance)
  • No timing envelope, so “late” is invisible
  • “Allow because it might be needed”

Risk is the possibility of process impact from (a) absence of required behavior or (b) presence of unnecessary behavior. Threat narratives are optional. Process impact is not.

Risk statement format
If <behavior deviation> occurs in <state>
then <impact> may occur within <time>
because <assumption violated / dependency>.
Impact classes (utilities)
  • False actuation
  • Missed actuation
  • Delayed actuation
  • Loss of visibility
  • Loss of controllability
  • Unsafe maintenance condition
Two-sided risk (don’t ignore absence)
In utilities, “nothing happened” can be the failure: missing trips, missing interlocks, missing alarms, missing time sync. Treat absence-of-required behavior as first-class risk.
Output
Risk register derived from behavior inventory, with explicit linkage back to assumptions and states.
Failure modes
  • Risk described as “attacker may…” without behavior
  • All risks treated as equal severity
  • Ignoring temporal bounds
  • Ignoring operator workflows

Before drawing zones, group assets and functions based on identical or tightly coupled risk profiles. This creates “proto-zones” without network bias.

Grouping criteria
  • Same function (protection/control/visibility/engineering)
  • Same failure impact and time window
  • Same trust assumptions
  • Same operational state constraints
Outputs
  • Risk-coherent groups
  • Group boundaries justified by impact containment
  • Group-to-behavior mappings
Why this matters
If you skip grouping, zones become topology art. Topology art fails the first time operations change a switch and nobody updates the diagram.

A zone is a set of functions sharing assumptions, behaviors, and risk characteristics. Zones exist to contain failure impact and constrain external behavior.

Zone definition checklist
  • Purpose (what functions it contains)
  • Shared assumptions
  • Allowed internal behaviors
  • External behaviors (must be minimal)
  • Failure containment target
Common utility zones (examples)
  • Process / IED bay zone
  • Station control zone
  • Station-to-control-center interface zone
  • Engineering / maintenance zone
  • Corporate IT zone (strictly separate)
Non-negotiable
“We put everything in one VLAN but we monitor” is not zoning. Monitoring is not containment.

Conduits are justified exceptions that allow specific behaviors to cross zones. Each conduit is a behavioral contract.

Conduit contract fields
  • Purpose (which behavior and why)
  • Directionality (who initiates)
  • State validity (when it is allowed)
  • Temporal bounds (how long / what window)
  • Failure consequence (what if misused)
  • Required evidence (logs, timings, approvals)
Conduit anti-patterns
  • Any-any rules “for operations”
  • Persistent engineering access in normal state
  • Bidirectional trust without reason
  • Conduits defined by network ports only
Utility reality
Maintenance is a different universe. Treat it as such: separate conduits, separate states, separate evidence.

Controls exist to prevent, detect, or limit the risks you already derived. If a control is not mapped to a risk statement and a conduit contract, it is noise.

Control mapping
  • Risk ID → Control objective
  • Control objective → Mechanism(s)
  • Mechanism → Evidence needed
  • Evidence → Assurance question
Control types
  • Preventive (gating, isolation, allow-lists)
  • Detective (state-aware monitoring, timing validation)
  • Corrective (safe fallback, rollback, isolation on anomaly)
  • Compensating (procedures, interlocks, two-person control)
Hard rule
If removing a security control breaks the plant, your security design violated engineering constraints. Controls must be additive and reversible, not structural dependencies.

Assurance is evidence that system behavior remains within bounds given declared assumptions. Assurance questions are the output of this model, not the input.

Evidence principles
  • State-aware (normal vs maintenance is not the same)
  • Time-aware (late is wrong even if “valid”)
  • Context-aware (who/why/under what workflow)
  • Reconstructable (post-event explainability)
Assurance outputs
  • Assurance question set
  • Evidence collection plan
  • Assumption validation schedule
  • Audit-ready traceability: assumption → behavior → risk → control → evidence
Operational test
If an operator asks “are we safe?” and the only answer is “we installed X”, you have no assurance. You have purchasing.

Traceability chain (mandatory)
Assumption → Behavior → Risk → Zone/Conduit → Control → Evidence → Assurance question
If any link is missing, you are making claims you cannot defend.
Assurance questions (derived outputs)

These questions are intentionally answerable and auditable. If your evidence cannot answer them, your system is not assurance-capable.

Assumptions
  • Which assumptions does this station/control center rely on today?
  • Which assumptions are validated continuously vs manually?
  • How do we detect assumption violation before process impact?
  • What happens when an assumption no longer holds?
Behavior
  • Are only required/acceptable behaviors present?
  • Are behaviors consistent with operational state?
  • Are timing/sequence bounds preserved?
  • Do we detect forbidden behaviors reliably?
Risk
  • Which risks are driven by missing required behavior?
  • Which risks are driven by unnecessary behavior?
  • Are time-to-impact and detection feasibility explicit?
  • Can we prove the control intent matches the risk statement?
Zones & Conduits
  • Are zones justified by shared risk and behavior (not topology)?
  • Are conduits minimal, directional, and state-bounded?
  • Are conduit exceptions time-limited and evidenced?
  • Can we explain every cross-zone behavior without hand-waving?
Controls
  • Is every control mapped to a risk statement?
  • Are controls bypassable without detection?
  • Do controls degrade safely (no plant breakage)?
  • Are compensating controls explicit where prevention is impossible?
Overall assurance
  • Can we explain why an action was allowed?
  • Can we reconstruct cause-effect post-event?
  • Can we prove the system operated within bounds during a disturbance?
  • Do we detect and respond when assumptions drift?
Practical endpoint
“Peace of mind” is not a feeling. It is the ability to answer these questions with evidence, fast, without improvisation.
